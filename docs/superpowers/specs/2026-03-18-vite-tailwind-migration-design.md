# Vite + Tailwind CSS Build Tool Migration — Design Spec

**Date:** 2026-03-18
**Status:** Draft
**Goal:** Replace the Tailwind CDN with a Vite + Tailwind CSS build pipeline, extract the shared config into a single file, and prepare the project for production deployment.

---

## Problem

The Nexopack website loads Tailwind CSS via a CDN `<script>` tag in both `index.html` and `products.html`. This approach has three issues:

1. **Performance:** The browser downloads the entire Tailwind library (~300KB) and generates styles at runtime. A build tool produces a purged CSS file containing only the classes in use — typically 5–15KB.
2. **Duplicated config:** The Tailwind configuration (colors, fonts, letter-spacing) is duplicated as an inline `<script>` in both HTML files. Changes must be made in two places.
3. **No dev tooling:** There is no local dev server, hot reload, or build step. The developer must manually refresh the browser after every change.

## Solution

Set up Vite as the build tool with Tailwind CSS v4 via the `@tailwindcss/vite` plugin. Define the shared Tailwind theme using the `@theme` directive in a single CSS entry point (`css/main.css`). Remove the CDN `<script>` tags from both HTML pages.

## Scope

### In Scope

- Initialize `package.json` with Vite, Tailwind CSS v4, and `@tailwindcss/vite`
- Create `vite.config.js` configured for multi-page setup (`index.html` + `products.html`)
- Create `css/main.css` as the single Tailwind entry point with `@theme` config (colors, fonts, letter-spacing) and imports for `css/home.css` and `css/styles.css`
- Update both HTML files: remove CDN `<script>` + inline config, add `<link>` to `css/main.css`
- Create `.gitignore` (node_modules, dist)
- Add `dev`, `build`, and `preview` scripts to `package.json`
- Verify both pages render identically after migration

### Out of Scope

- Hosting/deployment setup (Netlify, Vercel, etc.) — separate future task
- Adding new pages or content
- Changing the visual design
- JS bundling or module conversion — the existing `<script>` tags for `home.js` and `script.js` continue to work as-is

## CSS Entry Point and Theme Config

`css/main.css` serves as the single CSS entry point for both pages. In Tailwind CSS v4, the theme is configured directly in CSS using the `@theme` directive — there is no separate `tailwind.config.js` file. Content detection is automatic (Tailwind scans all project files).

```css
@import "tailwindcss";

@theme {
  --font-sans: 'DM Sans', sans-serif;
  --font-display: 'Playfair Display', serif;

  --color-sage-100: #DFE8E1;
  --color-sage-300: #98B69E;
  --color-sage-500: #6B8F78;
  --color-sage-700: #4A6B58;
  --color-sage-800: #395644;
  --color-sage-900: #243D2F;
  --color-cream: #F7F6F0;
  --color-cream-dark: #ECEAE3;
  --color-muted: #6D7873;
  --color-primary: #587E67;
  --color-divider: #d4ddd5;

  --tracking-spaced: 0.15em;
  --tracking-widest2: 0.25em;
}

@import "./home.css";
@import "./styles.css";
```

Both pages link to this file: `<link rel="stylesheet" href="/css/main.css">`.

Custom CSS is imported after `@import "tailwindcss"` so that custom selectors (`.hero-overlay`, `.product-card`, `.tab`, etc.) can override Tailwind utilities when needed. Both `home.css` and `styles.css` are imported globally — since their selectors are scoped to page-specific classes, there are no conflicts. This keeps the setup simple: one CSS entry point for all pages.

## Vite Config

```javascript
import { defineConfig } from 'vite'
import tailwindcss from '@tailwindcss/vite'

export default defineConfig({
  plugins: [tailwindcss()],
  build: {
    rollupOptions: {
      input: {
        main: 'index.html',
        products: 'products.html'
      }
    }
  }
})
```

When new pages are added, they are registered here (one line per page).

## HTML Changes

### What gets removed

From both pages:
- The CDN `<script src="https://cdn.tailwindcss.com"></script>` tag
- The inline `<script>tailwind.config = { ... }</script>` block (~25 lines)
- The separate `<link>` to page-specific CSS (`css/home.css` or `css/styles.css`) — now imported via `main.css`

### What gets added

To both pages:
- `<link rel="stylesheet" href="/css/main.css">`

### What stays unchanged

- Google Fonts `<link>` tags (preconnect + stylesheet) — these load fonts from Google's CDN and are not part of Tailwind
- JS files (`home.js`, `script.js`) continue to be loaded via `<script defer>` tags
- All other HTML content

## Developer Workflow

### Development

```bash
npm run dev
```

Opens a local server (typically `http://localhost:5173`). Changes to HTML, CSS, or JS files are reflected instantly in the browser without manual refresh.

### Production Build

```bash
npm run build
```

Outputs optimized files to `dist/`:
- HTML files with updated asset references
- A single purged CSS file (only classes in use)
- Copied static assets (images, icons)

### Preview Production Build

```bash
npm run preview
```

Serves the `dist/` folder locally to verify the production build before deploying.

## Constraints

- **Visual parity:** Both pages must look identical before and after migration. No design changes.
- **No JS changes:** `home.js` and `script.js` remain as standalone scripts. No module conversion.
- **Backwards compatible:** Adding new pages only requires creating the HTML file and adding one line to `vite.config.js`.
